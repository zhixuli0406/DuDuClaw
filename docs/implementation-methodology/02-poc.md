# Phase 2: PoC — 概念驗證執行

## 目標

在 1-2 週內完成概念驗證，用實際數據說服客戶進入正式導入階段。

## PoC 執行 SOP

### Day 1: 環境建置

- [ ] 確認硬體就緒（Mac Mini / NAS / VM）
- [ ] 安裝 DuDuClaw（`brew install duduclaw` 或 source build）
- [ ] 執行 `duduclaw onboard` 完成初始設定
- [ ] 執行 `duduclaw doctor` 確認環境健康
- [ ] 設定頻道（LINE OA / Telegram Bot token）

### Day 2-3: Agent 配置

- [ ] 選擇產業模板（`cp -r templates/<industry>/ ~/.duduclaw/agents/<name>/`）
- [ ] 客製化 SOUL.md（填入客戶品牌語氣、特定知識）
- [ ] 設定 CONTRACT.toml 邊界（依客戶需求調整）
- [ ] 匯入客戶資料（FAQ / 菜單 / 價目表）
- [ ] 設定 Evolution Engine 參數

### Day 4-5: 內部測試

- [ ] 執行 `duduclaw test` red-team 測試
- [ ] 內部團隊模擬客戶對話 50+ 輪
- [ ] 確認 CONTRACT.toml 邊界生效
- [ ] 確認多語系回覆品質
- [ ] 調整 SOUL.md 語氣與知識缺口

### Day 6-10: 試運行

- [ ] 連接正式頻道，開始接收真實客戶對話
- [ ] 每日檢視 Dashboard 統計
- [ ] 每日與客戶 15 分鐘 check-in
- [ ] 收集並處理 Edge Case
- [ ] 持續調校 SOUL.md

### Day 11-14: 結案評估

- [ ] 收集完整數據（見成效指標）
- [ ] 產出 PoC 結案報告
- [ ] 與客戶簡報結果
- [ ] 提出正式導入建議與報價

## 成效衡量指標

| 指標 | 計算方式 | PoC 目標 |
|------|---------|---------|
| 自動回覆率 | AI 獨立完成的對話 / 總對話 | > 70% |
| 平均回覆時間 | 收到訊息到回覆的時間 | < 5 秒 |
| 轉人工率 | 需要人工介入的對話 | < 30% |
| 零成本對話比 | 本地推論處理的對話 / 總對話 | > 50% |
| 客戶滿意度 | PoC 後問卷（1-5 分） | > 4.0 |
| CONTRACT 違規次數 | 違反邊界規則的回覆 | 0 |
