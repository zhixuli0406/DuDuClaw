# Phase 4: Pilot — 陪跑期監控與調校

## 目標

正式上線後 2-4 週密切監控，確保 Agent 穩定運行並持續優化。

## 陪跑期監控指標

### 每日監控（Dashboard 檢視）

| 指標 | 目標值 | 警戒值 | 行動 |
|------|--------|--------|------|
| 自動回覆率 | > 80% | < 60% | 檢視未覆蓋場景，更新 SOUL.md |
| 平均回覆時間 | < 5s | > 15s | 檢查推論延遲或網路問題 |
| 轉人工率 | < 20% | > 40% | 分析高轉人工場景，擴充知識 |
| 零成本對話比 | > 60% | < 30% | 調整 Evolution 參數或切換本地模型 |
| API 費用(日) | < 預算/30 | > 預算×1.5/30 | 檢查 Confidence Router 設定 |
| CONTRACT 違規 | 0 | > 0 | 立即修復邊界定義 |

### 每週調校流程

1. **資料檢視**（30 min）
   - 匯出本週對話統計
   - 辨識 Top 10 未覆蓋問題
   - 辨識 Top 5 轉人工原因

2. **SOUL.md 更新**（30 min）
   - 新增未覆蓋場景的知識
   - 調整語氣（依客戶回饋）
   - 更新過期資訊（促銷、價格）

3. **Evolution 檢視**（15 min）
   - 檢查 GVU 觸發次數與結果
   - 確認 SOUL.md 自動演化品質
   - 調整預測誤差閾值（如需）

4. **客戶 Check-in**（15 min）
   - 分享本週數據
   - 收集客戶回饋
   - 對齊下週優化方向

## Edge Case 處理流程

```
發現 Edge Case
    ↓
分類：
  ├─ 知識缺口 → 更新 SOUL.md / Memory
  ├─ 邊界不清 → 更新 CONTRACT.toml
  ├─ 語氣不當 → 調整 SOUL.md personality
  ├─ 技術問題 → 檢查 Agent 配置
  └─ 超出範圍 → 確認是否需要新場景
    ↓
測試修復（duduclaw test）
    ↓
部署更新
    ↓
追蹤 48hr 確認改善
```

## 陪跑期結案標準

- [ ] 連續 7 天自動回覆率 > 80%
- [ ] 連續 7 天無 CONTRACT 違規
- [ ] 客戶滿意度 > 4.0/5.0
- [ ] 所有已知 Edge Case 已處理
- [ ] 客戶端人員可獨立操作 Dashboard
- [ ] 緊急聯絡管道確認（LINE / 電話）
