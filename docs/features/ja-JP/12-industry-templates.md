# 業種テンプレートとOdoo ERP連携

> すぐに使えるビジネスインテリジェンス——数分でドメインエキスパートエージェントをデプロイ。

---

## たとえ話：レストランのセットメニュー vs アラカルト

新しいレストランを開く時、2つの選択肢があります：
- **アラカルト**：すべての料理をゼロから設計。最大の柔軟性、最大の労力。
- **セットメニュー**：実績のある組み合わせから始めて、好みに合わせてカスタマイズ。素早くローンチ、簡単にイテレーション。

DuDuClawの業種テンプレートはエージェントデプロイメントのセットメニューです。各テンプレートには特定の業種で運用するために必要なすべて——パーソナリティ、行動ルール、ドメイン知識——がすぐに使える状態で含まれています。

---

## 業種テンプレート

### テンプレートの内容

各テンプレートは完全なエージェントスターターキット：

```
templates/{industry}/
├── SOUL.md           # 業種に合わせたエージェントパーソナリティ
├── CONTRACT.toml     # 業種固有の行動境界
└── wiki/             # ドメインナレッジベース
    ├── glossary.md   # 業界用語
    ├── processes.md  # 標準運用手順
    └── compliance.md # 規制要件
```

**SOUL.md** — エージェントのパーソナリティが業界に適切なコミュニケーションスタイルで事前設定：
- 製造業エージェントは正確でメトリクス駆動の言語を使用
- レストランエージェントは温かくサービス志向で、食事関連の質問を自然に処理
- 貿易業エージェントは簡潔で数字重視、リスク意識が高い

**CONTRACT.toml** — 行動境界が業界規制を反映：
- 製造業エージェントは品質閾値に達しない材料を承認してはならない
- レストランエージェントは申告済みアレルゲンを持つ顧客に明示的な警告なしで料理を推薦してはならない
- 貿易業エージェントは投資関連の応答に常にリスク免責事項を含めなければならない

**Wiki** — エージェントが参照できるドメイン知識：業界用語と略語、標準運用手順、規制要件とコンプライアンスチェックリスト、一般的なシナリオと推奨応答。

### 利用可能なテンプレート

**製造業** — サプライチェーン管理、生産スケジューリング、品質管理、設備メンテナンスをカバー。

**レストラン** — オーダー管理、在庫追跡、顧客サービス、食品安全をカバー。

**貿易業** — 市場データ解釈、ポートフォリオ管理、リスク評価、コンプライアンスをカバー。

### カスタマイズフロー

テンプレートは出発点であり、拘束ではありません：

```
ステップ1：テンプレートをデプロイ
ステップ2：パーソナリティをカスタマイズ——SOUL.mdを自社のブランドボイスに合わせて編集
ステップ3：境界を調整——CONTRACT.tomlを特定のコンプライアンス要件に合わせて変更
ステップ4：ドメイン知識を追加——メニュー、サプライヤー、手順をwikiにインポート
ステップ5：進化に任せる——エージェントのパーソナリティがGVUサイクルで自己改善、
          カスタマイズされた規約境界内を維持
```

## Odoo ERP連携

### 課題

AIエージェントはビジネス運営について*話す*ことはできますが、*実行*できません。エージェントは顧客に請求書が必要だと知っていても、ERPシステムで作成できない——ブリッジがなければ。

### ソリューション

DuDuClawには、エージェントを世界で最も広く使われているオープンソースERPシステムの1つであるOdooに直接接続するミドルウェアが含まれています：

```
ユーザー：「顧客ABCの販売注文を作成、Widget X 10個」
     |
     v
エージェントが意図を理解
     |
     v
MCPツール呼び出し：sale_order_create
     |
     v
DuDuClaw Odoo BridgeがJSON-RPC呼び出しに変換
     |
     v
Odoo ERPシステムが販売注文を作成
     |
     v
Bridgeが結果を返却（注文番号、合計金額）
     |
     v
エージェント：「販売注文SO-2024-0042を作成。顧客ABC。
               Widget X 10個、合計：$1,500。」
```

### 利用可能な操作（15 MCPツール）

ブリッジは4つのビジネスドメインの操作を公開：

**CRM** — リード資格認定、商談作成、リードステータス更新
**販売** — 販売注文作成、注文ステータス確認、見積生成
**在庫** — 在庫レベル確認、在庫数量調整、出荷追跡
**会計** — 販売注文から請求書作成、支払処理、残高照会

### エディション検出

Odooは2つのエディション：Community Edition（CE、オープンソース）とEnterprise Edition（EE、有料）。一部の機能はEEでのみ利用可能。

ブリッジはこれを自動的に処理：初回接続時にOdooエディション（CEまたはEE）を検出し、検出されたエディションがサポートするMCPツールのみを公開。設定不要——ブリッジがOdooインスタンスを調査して自動適応します。

### イベント同期

操作の実行に加え、ブリッジはOdoo内のイベントをリッスンすることもできます：

```
Odooイベント発生：
  - 新リード作成
  - 注文ステータス変更
  - 請求書期限超過
     |
     v
イベントポーリングが変更をキャッチ
     |
     v
エージェントが通知を受信
     |
     v
エージェントがアクションを取れる：
  - 新リードを営業チームに通知
  - 顧客の注文ステータスを更新
  - 期限超過の請求書に支払リマインダーを送信
```

エージェントを受動的なツールユーザーから、実世界のイベントに反応するプロアクティブなビジネス参加者に変えます。

## テンプレート + ERPブリッジの組み合わせ

```
製造業テンプレート + Odoo Bridge：
  エージェントが在庫レベルを監視（Odoo）→
  重要材料の在庫不足を検出 →
  購買発注を自動作成 →
  設定チャネル経由で生産マネージャーに通知

レストランテンプレート + Odoo Bridge：
  エージェントが大口ケータリング注文を受信（チャネルメッセージ）→
  食材の在庫確認（Odoo在庫）→
  販売注文を作成（Odoo販売）→
  アレルゲンの懸念をフラグ（wikiナレッジ）→
  顧客に確認

貿易業テンプレート + Odoo Bridge：
  エージェントが市場データ更新を受信 →
  ポートフォリオポジションとクロスリファレンス（Odoo）→
  リスク閾値を超えるポジションを特定 →
  推奨アクション付きでトレーダーにアラート送信
```

## なぜ重要か

### タイムトゥバリューの短縮

テンプレートなしで業種固有のエージェントをデプロイするには：業界の用語とプロセスの調査、パーソナリティファイルの作成、行動境界の定義、ドメインナレッジベースの構築、テストとイテレーションが必要。テンプレートがあれば、これらのステップはすべて事前構築済み。数分でデプロイ可能。

### 運用の深さ

Odooブリッジがエージェントを会話アシスタントから運用ツールに変革。請求書の作成を*推奨する*だけでなく、*作成する*。AIアドバイスとビジネスアクションのギャップを埋めます。

### 標準化

テンプレートは業界ベストプラクティスをエンコード。テンプレートから作成されたエージェントは品質管理基準、安全プロトコル、在庫管理プラクティスをすでに理解。個々の運用者がこの知識を再発明する必要はありません。

### コンポーザビリティ

テンプレート、ERPブリッジ、進化システムがシームレスに連携。テンプレートが出発点を提供し、ERPブリッジが運用能力を提供し、進化システムが実世界のパフォーマンスに基づき継続改善——すべて規約の安全境界内で。

---

## 他システムとの連携

- **進化エンジン**：テンプレートからデプロイされたエージェントは他のエージェントと同様に進化。テンプレートは出発点であり恒久的状態ではない。
- **行動規約**：各テンプレートに業界コンプライアンス要件に合わせた規約を含む。
- **メモリシステム**：wikiのドメイン知識がメモリシステムを通じてインデックスされ検索可能。
- **チャネル統合**：テンプレートエージェントは全7通信チャネルに対応。
- **コスト管理**：ERPブリッジ操作はCostTelemetryで追跡され予算が可視化。

---

## まとめ

業種テンプレートとOdoo ERPブリッジはエージェントデプロイメントの「ラストマイル」問題を解決します：汎用AIから、実世界で実際に*物事を行える*ドメインエキスパートへ。テンプレートが知識とパーソナリティを提供し、ERPブリッジが運用能力を提供し、進化システムが継続的な改善を保証します。
