## 二、學術論文精選

### 📄 arXiv:2605.02801 — RL for LLM-based Multi-Agent Systems through Orchestration Traces
*（May 4, 2026）*

- 定義 **Orchestration Traces**：包含子代理 spawning、delegation、communication、tool use、return、aggregation、**stopping decision** 的時序互動圖
- **關鍵發現**：目前沒有任何 RL 方法專門訓練 stopping decision（何時停止子代理）
- 對 DuDuClaw 意涵：收集 orchestration traces 是建立 RL feedback loop 的基礎，且 stopping 是最薄弱的環節

---

### 📄 arXiv:2601.13671 — The Orchestration of Multi-Agent Systems: Enterprise Adoption
*（May 2026）*

- 企業採用需要技術架構 + 組織協議雙重投入
- 未部署 orchestration 策略的組織「在生產負載下面臨系統脆弱風險」
- **2025 AI Agent Index**：30 個已部署 Agent 中，安全文件一致性嚴重不足

### 📄 arXiv:2604.23338 — Layered Attack Surface Framework for LLM-Based Agents
*（April 2026，v2）*

- 三大開放挑戰：
  1. **高維嵌入空間的異常偵測**（memory 層）
  2. **跨 Agent 共享記憶的存取控制粒度**
  3. **密碼學溯源認證（cryptographic provenance attestation）**
- 記憶成為主要攻擊面，防禦方僅 66% 的攻擊能被偵測

### 【提案 1】Sub-Agent Depth Governor（子代理深度治理器）
```
優先級: P1
靈感來源: Claude Code sub-agents 5層深度上線 + arXiv:2605.02801 stopping decision
問題陳述: Claude Code 已支援 sub-agents 最多 5 層遞迴生成，但缺乏費用監控與遞迴失控防護；
          DuDuClaw 若開放同等能力，需要 budget guard + depth limit + stopping signal
解決方案: 
  - 設定 max_depth 配置（預設 3，可調至 5）
  - 每層 sub-agent 繼承父 agent 的 remaining_budget（token 配額自動分割）
  - 實作 StoppingOracle：當 sub-agent 連續 3 輪無新 output 時自動終止
  - EvolutionEvents 記錄每次 depth 觸發事件，作為 orchestration trace 收集
競品對比: Claude Code 有 5 層但無費用分割；LangGraph 用 checkpoints 但無 budget inheritance
預期影響: 開放多層 Agent 能力同時控制費用爆炸風險，提升生產可靠性
實作複雜度: 中
```

---

### 【提案 2】Memory Multi-Signal Retrieval（三信號記憶檢索升級）
```
優先級: P1
靈感來源: Mem0 April 2026 算法升級（語意 + BM25 + Entity Matching）
問題陳述: DuDuClaw 目前記憶檢索依賴純語意相似度（vector similarity），
          時序查詢和多跳推理準確率偏低，特別是跨任務長期記憶的召回
解決方案:
  - 在記憶檢索層加入 BM25 關鍵字匹配作為第二信號
  - add() 時自動提取 entity（人名、地名、概念）並建立 entity index
  - 三信號融合：weighted combination（α×semantic + β×BM25 + γ×entity_match）
  - Agent-generated facts 與 user-stated facts 同等優先級儲存
競品對比: Mem0 時序查詢 +29.6 points、多跳推理 +23.1 points（相對舊算法）；
          Letta 有 Conversations API 但無 multi-signal retrieval
預期影響: 直接強化 DuDuClaw 持久記憶的核心護城河；時序召回大幅提升
實作複雜度: 中（BM25 library 成熟，entity extraction 可複用 Mem0 開源方案）
```

### 【提案 3】Declarative Environment Blueprints（宣告式環境藍圖）
```
優先級: P1
靈感來源: Devin Desktop 廢棄 classic env → blueprints（June 30 deadline）
問題陳述: DuDuClaw agent 配置目前分散在多個設定文件，無法版本控制、
          無法一鍵重現環境，team 之間配置不一致
解決方案:
  - 定義 DuDuClaw Blueprint Schema（YAML/TOML）：agent identity、memory scope、
    tool permissions、model preference、evolution policy
  - Blueprint 版本控制：每次 EvolutionEvent 自動 snapshot blueprint 狀態
  - Blueprint diff viewer：比較兩個版本的 agent 配置差異
  - 支援 blueprint import/export（可與 Devin Desktop ACP 互操作）
競品對比: Devin blueprints（宣告式配置）；LangGraph 用 persistence backend；
          Cursor Enterprise 用 team-level governance
預期影響: 降低 onboarding 成本，提升配置可重現性，為 ACP bridge 打基礎
實作複雜度: 中
```

### 【提案 5】Orchestration Trace Collector（編排軌跡收集器）
```
優先級: P2
靈感來源: arXiv:2605.02801（RL for Multi-Agent through Orchestration Traces）
問題陳述: DuDuClaw 有 EvolutionEvents 記錄自演化事件，但缺乏系統化的
          orchestration trace 收集（sub-agent spawning、delegation 路徑、
          stopping 時機），無法作為未來 RL 訓練數據
解決方案:
  - 定義 OrchestrationTrace schema：7個事件類型
    （spawn、delegate、communicate、tool_use、return、aggregate、stop）
  - 每次 multi-agent session 自動生成 trace JSON
  - Trace quality scorer：標記 stopping decision 的合理性（是否太早/太晚）
  - 長期：trace 作為隱式 feedback 信號輸入 StoppingOracle（提案 1）
競品對比: 目前所有競品均無公開的 orchestration trace RL 訓練（arXiv 確認空白）；
          先進場者建立數據優勢
預期影響: 建立 DuDuClaw 獨有的 orchestration 訓練數據集，6-12 個月後轉化為能力優勢
實作複雜度: 低（schema 設計 + 現有 EvolutionEvents 擴充）
```
