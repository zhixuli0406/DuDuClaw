## 三、技術決策（ADR W24-02）

### TD-2026-0520-01：DuDuClaw 執行引擎策略 vs LangGraph 1.2.0
**背景**：PM 報告指出 LangGraph 1.2.0 已具備 DuDuClaw 規劃中的 Timeout / Checkpoint / Hooks 等可靠性功能，技術媒體定位其為「production AI agent 預設框架」。Agnes 問題 1 需 TL 給出技術立場。

**TL 評估**：

| 選項 | 優點 | 缺點 |
|------|------|------|
| LangGraph 作為執行層基礎 | 跟上功能進度快、社群生態豐富 | Vendor 依賴、無法控制 roadmap、與 DuDuClaw 架構哲學衝突 |
| 維持獨立引擎（參考 LangGraph 設計） | 架構彈性、差異化空間、可控 | 需自主維護可靠性原語 |

**TL 裁定**：**維持 DuDuClaw 獨立執行引擎**，但採用「LangGraph API 語義相容」策略：
- TimeoutPolicy / NodeTimeoutError / GracefulShutdown 語義與 LangGraph 1.2.0 對齊
- 內部實作完全獨立，保持 DuDuClaw 架構控制權
- 差異化聚焦：記憶系統（Tiered + Temporal Decay）、Orchestration Trace、Identity Graph
- 記錄為 ADR-004：Agent Execution Engine Strategy

**理由**：DuDuClaw 的核心護城河在記憶系統與 Identity 層，執行引擎標準化可降低遷移成本，但完全依賴 LangGraph 會失去差異化空間與架構彈性。
