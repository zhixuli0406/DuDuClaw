# v1.19.0 實作規格 — 記憶智能化

> **主題**：把 W18/W19 設計但尚未實作的「記憶智能層」收斂成一個連貫版本。
> **下個版本**：v1.18.0 → **v1.19.0**
> **三項範圍**：① Temporal Memory（地基）② Reflexion Loop MVP（核心價值）③ memory_fetch_batch（補完）

---

## 0. 架構落差更正（最重要）⚠️

W18/W19 設計文件成稿於記憶系統仍規劃為 **PostgreSQL + pgvector + Python MCP** 的時期。
實際上線的是 **Rust `SqliteMemoryEngine`**（`crates/duduclaw-memory/src/engine.rs`，SQLite + FTS5）。
本規格將所有過時假設更正如下，後續章節一律以「現行」為準：

| 設計文件舊假設 | 現行真實架構（v1.18.0） | 落地對應 |
|---|---|---|
| PostgreSQL `temporal_memory_nodes` 新表 | SQLite `memories` 表（既有） | **擴充欄位**，不建新表 |
| `metadata JSONB` 欄位 | `memories` **無 metadata 欄位** | 新增 `metadata TEXT DEFAULT '{}'`（存 JSON 字串） |
| `gin(metadata)` / `gin(tags)` 索引 | SQLite 無 GIN；`tags` 為 JSON 字串 | 以 FTS5 + 既有 `idx_memories_*` 索引；標籤過濾走 `tags LIKE` / 應用層 |
| pgvector 語義搜尋 | FTS5（`memories_fts`，unicode61） | 沿用 FTS5；語義向量為 v1.20+ 議題 |
| `WITH RECURSIVE` 取代鏈遍歷 | SQLite 同樣支援 `WITH RECURSIVE` | 直接沿用，無基建變更 |
| Python MCP stdio server（三端點） | **Rust CLI 已提供** `memory_search/store/read`（`crates/duduclaw-cli/src/mcp_memory_handlers.rs`），含 namespace（`ns_ctx`）+ quota（`daily_quota`）+ `Scope::MemoryRead/Write` | **放棄 Python 線**，全部落在 Rust CLI MCP |
| Neo4j / Apache AGE | 不引入 | MVP 零新基建 |

**結論**：本版**零額外基礎設施**，全部透過既有 SQLite migration loop（`engine.rs:386` 的 idempotent `ALTER TABLE` 模式）+ Rust MCP handler 擴充完成。

## 1. 範圍

### In scope（v1.19.0）
- **F1 Temporal Memory**：`memories` 加時態欄位 + 同 subject/predicate 自動衝突解決 + 搜尋預設只回有效記憶。
- **F2 Reflexion Loop MVP（橋接 `MistakeNotebook`）**：複用既有失敗記憶 → 召回注入答題 prompt（缺口）+ 同類 ≥3 固化成 semantic memory（綠地）。觸發複用 `ErrorCategory`（Significant/Critical），**非** GVU Verifier。
- **F3 memory_fetch_batch**：依 ID 陣列批次讀取（補 checkpoint 還原 / lazy-ref 的底層 API）。

### Out of scope（延後）
- quality_scorer_v2 + Intent Classifier（屬 skill-bank 信噪比，另立版本）。
- Python MCP stdio server（與 Rust CLI 重複，正式放棄）。
- 語義向量檢索 / Graph DB（Neo4j、Apache AGE）。
- Reflexion N5 數值欄位過濾（`critic_score` 範圍查詢）— 以 tag `score:low` 代替，v1.20 再評估。
- 二跳關係推理 / 完整 Zep TKG 對標。

## 2. F1 — Temporal Memory（地基）

### 2.1 Schema migration（沿用既有 idempotent 模式）
於 `engine.rs::init_tables` 的 `migrations` 陣列追加（每筆獨立 `ALTER`，重複欄位忽略）：

```rust
let migrations = [
    // ...既有 5 條...
    "ALTER TABLE memories ADD COLUMN valid_from TEXT",            // 預設 NULL → 讀取時 fallback = timestamp
    "ALTER TABLE memories ADD COLUMN valid_until TEXT",           // NULL = 仍有效
    "ALTER TABLE memories ADD COLUMN superseded_by TEXT",         // 指向取代它的 memory id
    "ALTER TABLE memories ADD COLUMN supersedes TEXT",            // 反向：被它取代的 id
    "ALTER TABLE memories ADD COLUMN subject TEXT",               // 三元組（衝突解決鍵）
    "ALTER TABLE memories ADD COLUMN predicate TEXT",
    "ALTER TABLE memories ADD COLUMN object TEXT",
    "ALTER TABLE memories ADD COLUMN confidence REAL NOT NULL DEFAULT 1.0",
    "ALTER TABLE memories ADD COLUMN metadata TEXT NOT NULL DEFAULT '{}'",  // 可選；F2b semantic 固化存來源 mistake id
];
```

新增索引（衝突解決查詢用，只索引當前有效記憶）：
```sql
CREATE INDEX IF NOT EXISTS idx_memories_triple
    ON memories(agent_id, subject, predicate) WHERE valid_until IS NULL;
CREATE INDEX IF NOT EXISTS idx_memories_valid
    ON memories(agent_id, valid_until);
```

**向後相容**：舊 DB 既有列 `valid_from/valid_until = NULL`；`row_to_entry` 對缺欄位 `unwrap_or` 容錯（沿用現有寫法）。讀取語義：`valid_from` 為 NULL 時視同 `timestamp`，`valid_until` 為 NULL 視同「仍有效」。

### 2.2 `MemoryEntry` 型別擴充
`crates/duduclaw-core/src/types.rs:1428` 加 `#[serde(default)]` 欄位（全部可選，保證舊 JSON 反序列化不破）：
```rust
#[serde(default)] pub valid_from: Option<DateTime<Utc>>,
#[serde(default)] pub valid_until: Option<DateTime<Utc>>,
#[serde(default)] pub superseded_by: Option<String>,
#[serde(default)] pub supersedes: Option<String>,
#[serde(default)] pub subject: Option<String>,
#[serde(default)] pub predicate: Option<String>,
#[serde(default)] pub object: Option<String>,
#[serde(default = "default_confidence")] pub confidence: f64,   // 1.0
#[serde(default)] pub metadata: serde_json::Value,              // {}
```
同步更新 `SELECT_COLS`、`row_to_entry`、`store()` 的 INSERT 欄位列（`engine.rs:103 / 412 / 473`）。

### 2.3 自動衝突解決（opt-in）
僅當寫入 entry **同時帶 `subject` + `predicate`** 時觸發（一般對話記憶不帶，行為與現況完全相同）：
```
store_temporal(entry):
  if entry.subject.is_some() && entry.predicate.is_some():
      在同一 conn 交易內：
        UPDATE memories
          SET valid_until = entry.valid_from(或 now),
              superseded_by = entry.id
          WHERE agent_id = ? AND subject = ? AND predicate = ? AND valid_until IS NULL;
        entry.supersedes = <被更新那筆的 id>   // 取第一筆；多筆衝突記 warn
  INSERT entry (valid_from = entry.valid_from.unwrap_or(now), valid_until = entry.valid_until)
```
作為 `SqliteMemoryEngine` 新 inherent method `store_temporal()`；既有 `store()` 不動（trait 相容）。

### 2.4 搜尋預設過濾有效記憶
- `search()` / `search_layer()` 的 SQL 追加 `AND (m.valid_until IS NULL OR m.valid_until > datetime('now'))`。
- 新增可選參數路徑 `include_historical: bool`（預設 false）。為不破 `MemoryEngine` trait，以新 inherent method `search_filtered(agent_id, query, opts)` 承載，trait `search()` 內部呼叫它帶預設值。

### 2.5 時態查詢（MVP 最小集）
- `get_history(agent_id, subject, predicate) -> Vec<MemoryEntry>`：沿取代鏈，用 `WITH RECURSIVE` 或單純 `ORDER BY valid_from`。
- Point-in-time（`valid_from <= t AND (valid_until IS NULL OR valid_until > t)`）：MVP 先做 engine method，**不**急著開 MCP 參數，v1.20 再對外。

### 2.6 F1 任務拆解
| # | 任務 | 估時 | 依賴 |
|---|---|---|---|
| F1-1 | migration 9 欄 + 2 索引 + 既有 DB 升級測試 | 0.5d | — |
| F1-2 | `MemoryEntry` 欄位 + `SELECT_COLS`/`row_to_entry`/`store` 同步 | 0.5d | F1-1 |
| F1-3 | `store_temporal()` 衝突解決（交易內 supersede） | 1d | F1-2 |

| | **小計** | **~4.5d** | |

## 3. F2 — Reflexion Loop MVP（橋接 MistakeNotebook，~2.5d）

> **2026-06-15 查證更正**：原設計假設「接 GVU Verifier 寫新 reflection_note」皆不成立，已查清如下：
> 1. **GVU Verifier 不是 critic**：GVU（`gvu/`）驗的是 **SOUL.md 提案**（`VerificationResult::Approved{confidence}`，0–1，≥0.7）。
> 2. **真正的任務品質訊號 = `PredictionError.composite_error` / `ErrorCategory`**（`prediction/engine.rs:44`，0–1，Negligible/Moderate/Significant/Critical，門檻由 MetaCognition 自適應）。**不要 hardcode 0.70**。
> 3. **`MistakeNotebook`（`gvu/mistake_notebook.rs`）已是半套 Reflexion**：記錄具體失敗（user/agent/錯因/ground truth）、`MistakeCategory`、SQLite 持久化、FIFO 50/agent、`resolved` 旗標、`query_by_agent/query_by_topic` 召回。
> 4. **episodic→semantic consolidation 不存在**（gateway 的 consolidation 全是 SOUL.md `SoulPatchOp::Consolidate`）。
>
> → 本節改為「**橋接既有 + 補兩個缺口**」，不新建平行 reflection_note 儲存。

### 3.1 既有可複用（不重做）
- **記錄**：`channel_reply.rs:1744` `build_mistake_entry` 已在任務失敗時寫入 MistakeNotebook（`Coding→Capability` / `QA→Factual` / else `Behavioral`）。
- **觸發**：`ErrorCategory ∈ {Significant, Critical}`（自適應門檻）已驅動下游演化；reflexion 沿用，**不新增門檻碼**。
- **召回 API**：`query_by_agent` / `query_by_topic`（keyword overlap，0 LLM 成本）。

### 3.2 本版新增缺口
- **F2a 召回回注答題 prompt（最高 CP 值）**：目前 mistake 召回（`channel_reply.rs:2119` `query_by_agent`）**只餵 GVU Generator 改 SOUL.md，沒回注答題**。新增：在既有記憶注入點（`channel_reply.rs:1034`，`search_facts` 旁）以 `query_by_topic(agent, user_query, 3)` 召回近期未解決 mistake，注入 system context：
  ```
  [Past Mistakes to Avoid]
  1. {what_went_wrong} → {corrective_insight}（category: {cat}）
  ```
  這就是「跨任務漸進式學習」的直接交付。
- **F2b episodic→semantic 固化（綠地）**：同一 `MistakeCategory` 未解決累積 **≥3** → 抽通用規則寫入 `memories`（`layer=semantic`、`source_event="reflexion_consolidation"`），用 F1 取代鏈關聯來源、並把來源 mistake 標 `resolved=true`。semantic 規則之後成為長期召回來源（取代逐筆 episodic 噪聲）。

### 3.3 F2 任務拆解
| # | 任務 | 估時 | 依賴 |
|---|---|---|---|
| F2a | `query_by_topic` 召回 + 答題 prompt 注入（`channel_reply.rs:1034` 旁） | 0.5d | — |
| F2b | `count_unresolved_by_category` + ≥3 固化成 semantic（F1 取代鏈 + 標 resolved） | 1.5d | F1-3 |
| F2-test | 整合測試（召回命中、≥3 觸發固化、不復發走 resolved） | 0.5d | F2a,b |
| | **小計** | **~2.5d** | |

> 移除（相對原設計）：memories 新建 reflexion_note、`failure_mode` 新枚舉、tag/metadata-based reflexion、30d TTL 平行儲存、GVU Verifier 掛點。改用 MistakeNotebook 既有 `MistakeCategory` + `resolved` 生命週期。

## 4. F3 — memory_fetch_batch（補完）

### 4.1 Engine method
`SqliteMemoryEngine::get_by_ids(agent_id, ids: &[String]) -> Result<Vec<MemoryEntry>>`
- `SELECT ... WHERE id IN (?,?,...) AND agent_id = ?`（動態綁定佔位符；上限 100）。
- Agent 隔離同 `get_by_id`（跨 agent id 不回傳，不洩漏存在性）。

### 4.2 MCP 工具

## 5. 排程與依賴

```
里程碑 M1（地基）：F1-1 → F1-2 →（F1-3, F1-4 並行）         ┐
里程碑 M1'（並行）：F3-1 → F3-2 → F3-3（完全獨立，可同週交付）┘
里程碑 M2（智能）：F2a（獨立，可早交付）；F2b（依 F1-3）

```
- **關鍵路徑**：F1-2 → F1-3 → F2b（Temporal 取代鏈是 semantic 固化的前置）。
- **可先行交付**：F3（batch）、F2a（mistake 召回注入）皆與 F1 schema 互不阻塞，建議第一週先合拿綠燈。
- **總估時**：約 **8.5 人日**（F1 4.5 + F2 2.5 + F3 1.5），符合 minor 版規模（F2 因橋接既有從 5d 降為 2.5d）。

## 6. 測試策略
- 覆蓋率目標 **≥ 80%**（沿用專案門檻）；engine 層維持現有 in-memory 測試風格（`SqliteMemoryEngine::in_memory()`）。
- 必測邊界：① 舊 DB 升級（無新欄位 → 正常讀寫）② 同 subject/predicate 重寫 → 舊筆 `valid_until` 落、`superseded_by` 正確 ③ 搜尋預設不回過期、`include_historical` 回全部 ④ 跨 agent 隔離（temporal / batch 皆需）⑤ reflexion 召回只回 30d 內 ⑥ consolidation 第 3 筆觸發、升 semantic ⑦ batch partial-hit 不報錯。
- 整合：GVU 低分 → 一輪 reflexion 寫入 → 下次同 task_type 召回命中。

## 7. 風險與緩解
| 風險 | 緩解 |
|---|---|
| 觸發訊號指錯層（GVU vs Prediction） | **已查證**：用 `ErrorCategory`（Significant/Critical），非 GVU Verifier；門檻沿用 MetaCognition 自適應 |
| 衝突解決誤殺（subject/predicate 撞庫） | opt-in：僅帶三元組才觸發；多筆衝突記 warn 不靜默 |
| 兩套失敗儲存不一致（MistakeNotebook vs memories） | F2 橋接既有 MistakeNotebook，不新建平行 reflection store；semantic 只存「固化後」通用規則 |
| mistake 內容注入破壞 prompt | 注入前沿用既有 XML escape（`call_claude_summarize` 同模式） |
| 召回污染 context（過時策略） | 召回限未解決（`resolved=false`）+ limit 3；固化後來源標 resolved |

## 8. 不納入本版（明確延後）
quality_scorer_v2 / Intent Classifier、Python MCP stdio server、語義向量、Graph DB、Reflexion N5 數值過濾、point-in-time 對外 MCP 參數、MistakeNotebook 遷入 `memories` 表（兩套儲存合一留待 v1.20 評估）。
